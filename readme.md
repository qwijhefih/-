# 🎓 정보처리산업기사 AI 맞춤형 학습 플랫폼

> **Gemini API 기반의 무한 문제 생성 및 1:1 AI 튜터링 서비스**

### 1. 프로젝트 개요
정보처리산업기사 자격증 취득을 목표로 하는 학습자를 위한 웹 애플리케이션입니다. 기존의 정적인 기출문제 풀이 방식에서 벗어나, **생성형 AI(Google Gemini)**를 활용하여 실시간으로 부족한 개념의 문제를 생성하고, 대화형 튜터를 통해 심도 있는 학습이 가능하도록 구현했습니다.

### 4. ✨ 핵심 기능
* **🤖 AI 문제 자동 생성:** 사용자가 원하는 카테고리(운영체제, DB 등)와 난이도를 선택하면 AI가 새로운 문제를 실시간으로 출제합니다.
* **💬 문맥 인식 AI 튜터:** 단순 질의응답이 아닌, 이전 대화 맥락을 기억하는 챗봇을 구현하여 꼬리에 꼬리를 무는 학습이 가능합니다.
* **📊 학습 통계 & 약점 분석:** 사용자의 풀이 데이터를 분석하여 카테고리별 정답률과 취약점을 시각화된 차트로 제공합니다.
* **🔁 간격 반복 학습:** 틀린 문제는 에빙하우스 망각 곡선 이론을 적용한 알고리즘을 통해 최적의 주기로 재출제됩니다.
* **💻 실전 코딩 테스트 환경:** `highlight.js`를 적용한 코드 가독성 향상 및 실전과 동일한 타이머 모드를 지원합니다.

### 5. 🚀 트러블 슈팅 (문제 해결 과정)
1. AI 챗봇의 문맥(Context) 단절 문제 해결
문제상황: 사용자가 "이거 더 자세히 설명해줘"라고 질문했을 때, AI가 '이거'가 무엇인지 인지하지 못하고 동문서답하는 현상이 발생했습니다. Gemini API가 상태를 저장하지 않는 Stateless 방식이라 이전 대화 맥락이 유지되지 않았기 때문입니다.

해결방안:

프론트엔드: 대화 내용을 history 배열 객체로 관리하여 브라우저 세션 동안 누적했습니다.

데이터 통신: API 요청 시 단순 쿼리만 보내는 것이 아니라, 누적된 history 전체를 백엔드로 전송하도록 로직을 변경했습니다.

백엔드: 수신한 데이터를 LLM이 이해할 수 있는 user ↔ model 구조로 재구성하여 요청을 보냈습니다.

결과: 이전 대화의 맥락을 완벽하게 기억하고 답변하는 챗봇을 구현하여, 끊김 없는 학습 경험을 제공했습니다.

2. 생성형 AI 응답 파싱 오류 방어 (방어적 코딩)
문제상황: AI에게 "JSON 형식으로만 답해줘"라고 프롬프트를 입력해도, 간혹 마크다운 코드 블록(```json)이나 잡담("여기 결과입니다")을 섞어서 응답하는 경우가 발생했습니다. 이로 인해 json.loads() 실행 시 JSONDecodeError가 발생하여 서버가 중단되었습니다.

해결방안: 응답 데이터를 즉시 파싱하지 않고, 후처리(Post-processing) 파이프라인을 구축했습니다. strip()으로 공백을 제거하고, startswith 등을 활용해 JSON과 무관한 마크다운 문법을 제거한 뒤 순수 JSON 문자열만 추출하여 파싱하도록 예외 처리를 강화했습니다.

결과: 외부 API의 변칙적인 응답에도 서버가 멈추지 않고 안정적으로 문제 생성 서비스를 제공할 수 있게 되었습니다.

3. 학습 효율을 위한 '간격 반복 알고리즘' 구현
문제상황: 단순히 틀린 문제를 쌓아두기만 하는 방식은, 학습 데이터가 늘어날수록 복습해야 할 양이 감당할 수 없을 정도로 많아져 학습 효율을 떨어뜨렸습니다.

해결방안: 에빙하우스의 망각 곡선 이론을 적용한 '간격 반복(Spaced Repetition)' 알고리즘을 구현했습니다. 문제별로 review_count(복습 횟수) 필드를 만들고, 정답을 맞힐 때마다 재출제 간격을 1일 → 3일 → 7일 → 14일로 점진적으로 늘리도록 설계했습니다.

결과: 사용자가 '아는 문제는 천천히, 모르는 문제는 자주' 마주치게 하여 암기 효율을 극대화하고 학습 피로도를 낮췄습니다.

4. JSON 파일 I/O 성능 최적화 (데이터 리미팅)
문제상황: 별도의 DB 서버 없이 JSON 파일로 데이터를 관리하다 보니, 학습 이력(history)이 무한정 쌓일 경우 파일 크기가 커져 읽기/쓰기 속도가 느려지는 I/O 병목 현상이 우려되었습니다.

해결방안: 데이터를 저장하는 시점에 '슬라이딩 윈도우(Sliding Window)' 기법을 응용한 리미팅 로직을 적용했습니다. if len(history) > 20: 조건을 통해 최근 20건의 핵심 데이터만 유지하고, 오래된 데이터는 자동으로 소멸되도록 처리했습니다.

결과: 서비스 사용 기간이 길어져도 데이터 파일의 크기를 일정 수준으로 유지하여, 서버 메모리 부하를 방지하고 일정한 응답 속도를 보장했습니다.

5. 코드 문제 렌더링 시 XSS 보안 취약점 해결
문제상황: C언어 문제에 포함된 #include <stdio.h>와 같은 코드가 브라우저 렌더링 시 HTML 태그로 오인되어 화면에서 사라지거나, 악성 스크립트로 동작할 수 있는 XSS(크로스 사이트 스크립팅) 취약점이 발견되었습니다.

해결방안: 데이터를 DOM에 렌더링하기 직전에 실행되는 이스케이프(Escape) 유틸리티 함수를 구현했습니다. 특수문자(<, >, &)를 HTML 엔티티(&lt;, &gt;, &amp;)로 치환하여 코드가 실행되지 않고 텍스트로만 출력되도록 강제했습니다.

결과: 코드 문제의 가독성을 확보함과 동시에 웹 보안 위협을 원천 차단했습니다.
